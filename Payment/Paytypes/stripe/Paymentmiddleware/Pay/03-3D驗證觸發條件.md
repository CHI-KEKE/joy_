# Stripe 信用卡付款 - 3D 驗證觸發條件

## 3D 驗證觸發的兩層機制

Stripe 的 3D 驗證觸發分為兩層：

1. **我們的 Request 參數** → 控制 PaymentIntent 的行為傾向
2. **Stripe / 發卡行** → 最終決定是否真的需要 3D（SCA 法規、風險評估）

---

## 關鍵 Request 參數說明

| 參數名稱 | ExtendInfo 欄位 | 預設值 | 對 3D 的影響 |
|---------|---------------|--------|------------|
| `off_session` | `OffSession` | `false` | `true` = **嘗試跳過** 3D 驗證（靜默扣款） |
| `setup_future_usage` | 程式碼固定帶入 | 無 | `"off_session"` = **主動要求** 建立 3D 授權 mandate |
| `return_url` | `ReturnUrl` | 無 | 3D 驗證後的回跳 URL，**沒帶則 3D redirect 無法完成** |
| `three_d_secure_status_def` | `ThreeDSecureStatusDef` | 無 | ⚠️ **此欄位目前未被使用**（dead field） |

---

## 各情境的 3D 觸發分析

### 情境一：單純 Pay

**PaymentIntent 參數：**
```
confirmation_method = automatic
confirm = true
return_url = {提供的 return_url}
← 沒有 off_session
← 沒有 setup_future_usage
```

**特性：**
- `off_session` 欄位在 `strategy.Pay()` 路徑中**完全不被讀取**
- 是否觸發 3D **完全由 Stripe / 發卡行決定**
- 常見觸發條件：歐洲發卡行（SCA 合規）、高風險交易、銀行要求

**結果：**

| 情況 | 3D 觸發 |
|------|--------|
| Stripe 認為低風險 | ❌ 直接 `succeeded` |
| 銀行要求 SCA | ✅ `requires_action` + redirect URL |

---

### 情境二：記住信用卡（首次綁卡）

**PaymentIntent 額外參數：**
```
setup_future_usage = "off_session"    ← 程式碼固定帶入
off_session = "true"                  ← 僅當 ExtendInfo.OffSession = true 時帶入
```

**分析：**

| Request 帶的參數 | Stripe 行為 | 3D 觸發 |
|----------------|-----------|--------|
| `off_session = false`（預設） | `setup_future_usage=off_session` 通知 Stripe 建立 off-session mandate | ✅ **通常會觸發 3D** 完成授權設定 |
| `off_session = true` | 嘗試跳過，但 Stripe 仍可能因 SCA 強制要求 | ⚠️ 可能跳過，但不保證 |

**重要說明：**
- `setup_future_usage = "off_session"` 的意義是：「這張卡之後要用於無人在場的自動扣款」
- Stripe 為確保持卡人同意，**通常會要求 3D 驗證**來建立授權 mandate
- 3D 成功後，之後的 off-session 扣款才能正常進行

---

### 情境三：舊卡復用

**PaymentIntent 參數：**
```
payment_method = {已綁定的 Token}
off_session = "true"     ← 僅當 ExtendInfo.OffSession = true 時帶入
← 沒有 setup_future_usage
```

**分析：**

| Request 帶的參數 | Stripe 行為 | 3D 觸發 |
|----------------|-----------|--------|
| `off_session = true` | 告知 Stripe 持卡人不在場，直接靜默扣款 | ❌ **跳過 3D** |
| `off_session = false`（預設） | Stripe 依發卡行規則決定 | ⚠️ 可能觸發 3D |

**注意：** 帶 `off_session = true` 但卡片已過期或被撤銷授權時，Stripe 會回傳 ApiException（`3000`）。

---

## 三種情境對照表

| 情境 | `off_session` 欄位有效？ | `setup_future_usage` | 3D 觸發來源 |
|------|------------------------|---------------------|-----------|
| 單純 Pay | ❌ 不讀此欄位 | 無 | 純 Stripe / 發卡行決定 |
| 記住信用卡 | ✅（可嘗試跳過） | `"off_session"`（固定） | **主動觸發** mandate 授權 |
| 舊卡復用 | ✅（`true` = 跳過） | 無 | `off_session=false` 時由 Stripe 決定 |

## 3D 驗證完成後的後續流程

3D 驗證完成後，呼叫端需再呼叫 `QueryPayment` 確認最終狀態：

```
POST /api/v1/QueryPayment/CreditCardOnce_Stripe
        │
        ▼
GET /v1/payment_intents/{payment_intent_id}
        │
   ┌────┼──────────────────────────┐
   ▼    ▼                          ▼
succeeded  requires_payment_method  requires_action
  0000    + last_payment_error         2003
           → 3000（附 decline_code）
```

| QueryPayment 回應 status | ReturnCode | 說明 |
|------------------------|-----------|------|
| `succeeded` | `0000` | 3D 驗證通過，付款成功 |
| `requires_payment_method` + `last_payment_error` | `3000` | 付款被拒，附 `decline_code` |
| `requires_action` / `requires_confirmation` | `2003` | 仍在等待驗證 |
| `requires_payment_method`（無錯誤） | `2003` | 等待用戶輸入付款方式 |
| 其他未知狀態 | `9001` | UnhandledException（logger 記錄） |

## 整體 3D 驗證流程圖

```
strategy.Pay() / PaymentIntentAsync()
        │
        ▼
POST /v1/payment_intents
        │
   status = requires_action？
        │
   ┌────┴────┐
   ▼         ▼
  否         是
  │          │
succeeded  next_action.type
(0000)      │
        ┌───┴────────┐
        ▼            ▼
redirect_to_url  use_stripe_sdk
(瀏覽器導頁)     (Stripe JS SDK)
        │
   ReturnCode = 2003
   WebPaymentUrl = 3D 驗證 URL
        │
   用戶完成 3D 驗證
        │
   POST /api/v1/QueryPayment/...
        │
   GET /v1/payment_intents/{id}
        │
   確認最終 status
```
