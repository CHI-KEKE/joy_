匯入第三方支付資料
推進 SalesOrder / SalesOrderSlave / OrderSlaveFlow 狀態
依「付款方式 × 狀態 × 是否多付款」建立 PaymentRequest
補齊費率、撥款日、Gateway、Provider 等會計資訊

## ① 蒐集訂單與付款相關資料

讀取：

SalesOrder / SalesOrderGroup

OrderSlaveFlow

SalesOrderCustomInfo（JSON，處理多付款方式）

PayProfile / PayProfileProcessingFee

ConfigDB 設定

判斷：

第一付款方式 / 第二付款方式

是否為多付款（MultiPayment）

是否為小時達訂單（RetailStore）

是否為第三方支付類型


## ② 計算付款手續費（FeeRate）

依付款方式：

優先用 PayProfileProcessingFee

否則回退用 PayProfile

同時處理：

第一付款方式

第二付款方式（若有）

## ③ 寫入「第三方支付單」

條件成立時才會做

從
SourceTradesOrderThirdPartyPayment

複製資料到
SalesOrderThirdPartyPayment

同時取得：

第三方支付狀態（Success / AuthSuccess / Cancel…）

交易編號、金額、資訊

## ④ 依第三方支付狀態，更新訂單狀態

更新表：

SalesOrderSlave
SalesOrderSlaveDateInfo
OrderSlaveFlow

狀態範例：

Success / AuthSuccess → WaitingToShipping
Cancel / Timeout → Cancel
WaitingToPay → WaitingToPay

## ⑤ 建立請款單（PaymentRequest）

這是整支 CSP 最主要的產出之一

依以下條件判斷是否建立：

付款方式（LinePay / PXPay / CreditCard / Wallet / …）

第三方支付狀態

是否為多付款

是否為 ATM / 現金（有些不建請款單）

建立時會設定：

請款狀態（Finish / WaitingToPaymentRequest / PaymentRequesting）

請款金額

確認日（ConfirmDate）

預計撥款日（ExpectedFundingDate）

手續費（Fee、FeeRate）

Gateway / PaymentServiceProvider

## ⑥ 處理「多付款方式」情境

第一付款方式 + 第二付款方式

可能：

建一張請款單
或建多張（各付款方式一張）

特例處理：

StoreCredit
HamiPoint
EasyWallet
ATM / CustomOfflinePayment

## ⑦ 全程包在 Transaction 中

發生錯誤會：

Rollback
回傳「第三方支付轉單失敗」






## ② 判斷「付款情境」

#### A. 第一付款方式 = 第三方支付

建：

SalesOrderThirdPartyPayment
PaymentRequest（1 或多張）

更新 SalesOrderSlave / OrderSlaveFlow

#### B. 第一付款方式 = 現金 / 信用卡，第二付款方式 = 第三方

➡️ 只針對第二付款方式

建 ThirdPartyPayment
視條件建 PaymentRequest

#### C. 特例（ATM / CustomOfflinePayment）

➡️ 只建 ThirdPartyPayment，不建請款單




## ③ PaymentRequest 建立邏輯（這支最複雜的地方）

你這裡實際上是在做：

付款方式 × 狀態 → 
  ConfirmDate / ExpectedFundingDate /
  Status / IsClosed / Provider / Fee

而且：

每一種 PayType 都有自己的一段 IF ELSE

MultiPayment 會再多跑一輪（甚至 while loop）

👉 這一段是「業務規則集中地」，也是維護地獄
