# 回饋活動 — 維度分析

回饋活動的維度判斷分兩個層次：  
**第一層**：引擎 Rule 本身的 Scope 設定  
**第二層**：進引擎前的動態 Tag 注入

---

## 第一層：引擎 Rule Scope（`GetRuleObject` 組裝）

由 `PromotionEngineRuleBaseService.GetRuleObject()` 將維度寫入 Rule JSON，引擎計算 `IsMatch` 時使用：

| 維度 | Rule 欄位 | Point2 | RatePoint2 | Coupon |
|------|-----------|:------:|:----------:|:------:|
| 商品圈選（指定/排除 SalePage/Collection） | `IncludedProductScopes` / `ExcludedProductScopes` | ✅ | ✅ | ✅ |
| 門市圈選（指定 Location） | `IncludedLocationScopes` | ✅ | ✅ | ✅ |
| 會員對象（MemberTier / MemberCollection / All） | `MatchedUserScopes` | ✅ | ✅ | ✅ |
| 會員可視（曝光對象） | `VisibleUserScopes` | ✅ | ✅ | ✅ |
| 通路（Web / AppIOS / AppAndroid / InStore…） | `MatchedSalesChannels`（int flag） | ✅ | ✅ | ✅ |
| 地區（CountryProfileId） | `IncludeRegionScopes` | ✅ | ✅ | ✅ |
| 首購 | `ConsumerTags: ["FirstPurchase"]` | ✅ | ✅ | ✅ |
| 金流（PayProfileTypeDef） | `PaymentTypes` | ❌ | ❌ | ❌ |
| 物流（ShippingProfileTypeDef） | `ShippingTypes` | ❌ | ❌ | ❌ |
| 星期幾 | `MatchedSalesDayOfWeeks` | ❌ | ❌ | ❌ |

> 回饋活動繼承的 `RewardReachPriceRuleBaseService` 沒有 override `SetPaymentTypes`、`SetShippingTypes`、`SetMatchedSalesDayOfWeeks`，預設為 null，**不支援這三個維度**。

## 第二層：動態 Tag 注入（引擎計算前）

### 2-1 商品圈選（Collection）
- **執行時機**：`UpdateSalepageSkuItemListAsync` → `GetSalepageInfoListAsync`
- **機制**：打 Collection API（`GetMatchedSalepageCollectionAsync`），取每個 SalepageId 符合哪些 CollectionId
- **結果**：CollectionId 加入 `SalepageSkuItemEntity.Tags`
- **引擎比對**：Rule 的 `IncludedProductScopes`（TagProductScope）vs 商品 Tags

### 2-2 料號（OuterId）
- **執行時機**：`UpdateSalepageSkuItemListAsync` → `GetOuterIdTagsAsync`
- **兩條路徑**：
  - `PromotionTagOuterIdRepository`（DB 路徑，一般訂單計算）
  - `S3OuterIdRepository`（S3 路徑，菜籃計算）
- **只針對三種回饋活動類型**查詢
- **機制**：OuterId → PromotionTagId → 貼到商品 Tags
- **引擎比對**：Rule 的 `IncludedProductScopes`（TagProductScope）vs 商品 Tags

### 2-3 壽星（當月生日）
- **執行時機**：`GetRuleInfoListAsync` → `SetCustomRuleMemberCollectionToUserTagAsync`
- **觸發條件**：Rule JSON 中 `IsBirthdayMonthEnabled == true`
- **流程**：
  1. 取訂單時間月份（`context.Cart.Now.Month`）
  2. 查 DynamoDB（Key: `System_BirthdayMonth_{shopId}`）→ 取該月份的 CollectionId
  3. 打 Member Collection API（`PostMemberCollectionMatchAsync`）比對使用者
  4. 符合 → User.Tags 加入 `"CurrentBirthdayMonth"`
- **引擎比對**：Rule 的 `MatchedUserScopes`（`Tag: "CurrentBirthdayMonth"`）vs User.Tags

### 2-4 MemberCollection（自訂名單）
- **執行時機**：`GetMemberCollectionTagsAsync` → `SetMemberCollectionToUserTagAsync`
- **機制**：Rule 的 `MatchedUserScopes` 中含 GUID CollectionId → 打 Member Collection API → 符合則 CollectionId 加到 User.Tags
- **引擎比對**：Rule 的 `MatchedUserScopes`（TagUserScope）vs User.Tags

### 2-5 門市（Location）
- **執行時機**：`UpdateLocationTagsAsync`
- **BottomProcessGroup / PromotionBasketGroup**：空實作，LocationTags 始終為空（⚠️ 門市比對對線下訂單實際上不生效）
- **CustomRuleBasketGroup**（點數菜籃）：從 S3 取門市 JSON，比對 `LocationId` → 轉換為 `TagId` → 填入 `LocationTags`
- **引擎比對**：Rule 的 `IncludedLocationScopes`（TagLocationScope）vs LocationTags；線上訂單 `isLocationAlwaysMatch = true` 恆通過

## 維度與 Lack 欄位對應

| 維度 | 未達成時的 Lack 欄位 |
|------|---------------------|
| 金額不足 | `LackOfPrice` |
| 會員等級/Collection/壽星/首購 | `LackOfUserScope` |
| 商品不在圈選範圍 | `LackOfProduct` |
| 通路不符 | `LackSalesChannel` |
| 地區不符 | `LackOfRegion` |
| 物流不符 | `LackOfShipping` |
| 星期幾不符 | `LackOfDateTime` |
