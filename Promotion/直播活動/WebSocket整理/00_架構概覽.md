# WebSocket 架構概覽

## 為什麼要實作 WebSocket

1. **即時性要求**
   直播間需要即時推送留言狀態（如加車、抽獎結果），HTTP 輪詢會造成延遲和資源浪費，WebSocket 提供持久連線，訊息即時推送。

2. **高平行處理需求**
   一個直播間可能有多個連線同時存在，需要訊息廣播功能。

3. **降低伺服器負載**
   避免頻繁 HTTP 請求建立/斷開連線，減少 Header 重複傳輸的開銷，提升使用者體驗。

---

## 技術選擇：AWS API Gateway WebSocket

此專案不使用傳統的 SignalR，而是採用 AWS API Gateway WebSocket：

- ✅ **無伺服器架構**：不需要管理 WebSocket 伺服器
- ✅ **自動擴展**：AWS 自動處理連線擴展
- ✅ **區域化部署**：支援多市場（TW/HK/MY）的區域部署
- ✅ **分離前後台連線**：Frontend API 和 Backend API 分開管理

### AWS API Gateway vs 自建 SignalR 比較

| 面向 | AWS API Gateway WebSocket | 自建 SignalR Server |
|------|--------------------------|-------------------|
| 伺服器管理 | ✅ 無需管理，完全託管 | ❌ 需要管理伺服器、負載平衡器 |
| 自動擴展 | ✅ AWS 自動處理，無限擴展 | ⚠️ 需要手動設定 Auto Scaling |
| 高可用性 | ✅ AWS 保證 99.95% SLA | ⚠️ 需要自己實作 Multi-AZ |
| 區域部署 | ✅ 輕鬆部署到多個 Region | ❌ 需要在各區域建立基礎設施 |
| 連線狀態管理 | ⚠️ 需要自己用 Redis 管理 | ✅ SignalR 內建連線管理 |
| 成本（小流量） | ✅ 按使用量計費，沒流量不收費 | ❌ 需要至少一台伺服器持續運行 |
| 成本（大流量） | ⚠️ 可能昂貴（每百萬連線分鐘） | ✅ 固定成本 |
| 延遲 | ⚠️ 多一層 API Gateway（~50ms） | ✅ 直連伺服器，延遲更低 |
| 開發複雜度 | ⚠️ 需要自己實作訊息路由邏輯 | ✅ SignalR 抽象層簡化開發 |
| 監控與日誌 | ✅ 整合 CloudWatch，完整監控 | ⚠️ 需要自建監控系統 |
| 前後台分離 | ✅ 輕鬆建立多個 API | ⚠️ 需要程式碼層級控制 |

## 為什麼 WebSocket 要拆分 Frontend 和 Backend API？

### 🔐 安全性隔離

若只用單一 API，URL 暴露在前端原始碼中，惡意使用者可以直接存取並嘗試偽造請求或 DDoS 攻擊。

拆分後：
- 消費者只知道 Frontend API URL（公開）
- Backend API URL 不公開，即使駭客攻擊 Frontend API，Backend API 不受影響
- 可以針對 Frontend API 設定更嚴格的 Rate Limit
- 可以分別監控兩個 API 的流量和錯誤率

### 📊 權限控制與資料隔離

- **Frontend API（消費者專用）**：Rate Limit 較低、CloudWatch Alarm 監控異常流量、基礎設施層級保護
- **Backend API（內部/系統專用）**：Rate Limit 較高、獨立監控和告警

## 為什麼需要 ConnectionId？

同一個 `liveSessionId` 下可能同時有多個連線（多裝置、多使用者）。  
如果只用 `liveSessionId` 當識別，斷線時無法判斷要刪除哪一條連線。

AWS API Gateway 每次建立 WebSocket 連線時會產生唯一的 `connectionId`：

- **唯一性**：全域唯一，格式類似 `L0SM9cOFvHcCJhg=`（Base64）
- **生命週期**：連線建立時產生，斷線後失效，無法重複使用
- **與實體的關係**：
  - 1 個使用者 可以有 N 個 connectionId（多裝置）
  - 1 個直播間 可以有 N 個 connectionId（多人）
  - 1 個 connectionId 只對應 1 個 WebSocket 連線

```csharp
// 斷線時必須使用 connectionId
public async Task DisconnectAsync(string connectionId)
{
    var connectionCacheEntity = await _liveSessionConnectionService
        .GetConnectionCacheEntityAsync(connectionId);
    
    var liveSessionId = connectionCacheEntity.LiveSessionId;
    
    // 只移除這一個 connectionId，不影響其他連線
    await _liveSessionConnectionService
        .RemoveLiveSessionConnectionIdsAsync(liveSessionId, new List<string> { connectionId });
    
    await _liveSessionConnectionService
        .RemoveConnectionCacheEntityAsync(connectionId);
}
```

## 路由自己實作的意思（vs SignalR）

### SignalR 的自動路由（框架提供）

```csharp
public class ChatHub : Hub
{
    public async Task JoinGroup(string liveSessionId)
    {
        // 框架自動管理 Group
        await Groups.AddToGroupAsync(Context.ConnectionId, $"LiveSession_{liveSessionId}");
    }

    public async Task SendMessage(string liveSessionId, string msg)
    {
        // 框架自動路由到 Group 內所有連線，不需要查詢誰在這個 Group
        await Clients.Group($"LiveSession_{liveSessionId}").SendAsync("ReceiveMessage", msg);
    }
}
```

### AWS API Gateway + 自己實作路由（此專案）

1. **定義 Group 單位**：以 `liveSessionId` 為 Group 單位
2. **自己用 Redis 管理 Group 成員**：`SADD websocket:liveSession:{liveSessionId}:connections {connectionId}`
3. **自己從 Redis 查詢並廣播**：取出所有 connectionId，逐一呼叫 `PostToConnectionAsync`
4. **自己處理發送失敗**：捕捉失敗的 connectionId，廣播後統一清理（Lazy Cleanup）

> 詳細的 Redis 資料結構與 TTL 設計見 `02_連線管理.md`。
