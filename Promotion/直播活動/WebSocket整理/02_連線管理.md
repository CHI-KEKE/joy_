# WebSocket 連線管理

## 1️⃣ Redis 資料結構：LiveSession → ConnectionIds（Redis Set）

```bash
LiveSessionId
  └── Redis Set
        ├── connectionId A
        ├── connectionId B
        └── connectionId C
```

每一條 WebSocket 連線在 Redis 中對應兩份資料：

```
# 單一連線資訊
websocket:connection:{connectionId}        → 存：shopId、liveSessionId、platformUserId、userName

# 直播間的所有連線集合（Set）
websocket:liveSession:{liveSessionId}:connections  → 存：所有 connectionId
```

- 使用 Set：天然避免重複
- 用 LiveSession 當廣播群組（合理）

廣播時的流程：
1. 從 `liveSession Set` 取出所有 `connectionId`
2. 逐一對各 `connectionId` 推送訊息

---

## 2️⃣ 避免重複 ConnectionId

### Redis Set 天然去重

Redis 的 `SADD` 指令（對應 `SetAddAsync`）具有天然去重特性，Set 集合本身不允許重複元素。  
如果嘗試加入已存在的 member，操作會回傳 `false` 且不會改變集合內容。

```
Key:     websocket:liveSession:{LiveSessionId}:connections
Command: SADD
行為:    自動去重，重複加入會回傳 0 (false)
```

### SET NX 確保連線不重複建立

建立單一連線快取時使用 `SET NX (Not Exists)`，搭配 Redis Set 的原子操作避免併發問題。

**情境 1：相同 ConnectionId 同時建立連線**

```
時間 T1: 用戶 A 呼叫 HandleConnectionAsync(connectionId: "abc123")
時間 T1: 用戶 A 呼叫 HandleConnectionAsync(connectionId: "abc123")  ← 重複請求
↓
Redis SET NX 原子操作:
  - 第一次請求: SET 成功，回傳 true
  - 第二次請求: Key 已存在，回傳 false
  - SADD 第一次: 成功加入 Set
  - SADD 第二次: 回傳 false（已存在）
結果: Set 中只有一個 "abc123"
```

**情境 2：網路延遲導致前端重試**

```
T1: 前端發起連線請求 1
T2: 前端因網路逾時重試，發起請求 2
↓
兩個請求都到達後端:
  - Redis Set 確保 ConnectionId 唯一
  - 最終 Set 只包含一個元素
  - 廣播時只會推送一次
```

## 3️⃣ 兩步寫入的非原子問題

建立連線時有兩個 Redis 操作：

```
1. SET websocket:connection:{connectionId}              (SET NX + TTL)
2. SADD websocket:liveSession:{liveSessionId}:connections
```

這兩步：
- ✔ 各自都是原子
- ❌ 合起來不是原子

如果在中間失敗（例如網路抖動），就會出現 **資料不一致（partial state）**：

| 失敗情境 | 結果 |
|---------|------|
| 步驟 1 成功、步驟 2 失敗 | 有 connection cache，但不在 liveSession set（廣播找不到此連線） |
| 步驟 2 成功、步驟 1 失敗 | Set 有 connectionId，但查不到連線資訊（殭屍）— 較少見但可能 |

> 目前透過「廣播前雙層驗證 / Lazy Cleanup」處理（見第 5 節）。

## 4️⃣ TTL 設計

| Redis Key | TTL |
|-----------|-----|
| `websocket:connection:{connectionId}` | 6 小時 |
| `websocket:liveSession:{liveSessionId}:connections` | 6 小時（**僅首次建立時設定，之後不刷新**） |

> 原本設計是個別連線 8 小時、LiveSession Set 6 小時，但 Set 比連線還短會造成「Set 已過期但快取還在，SendToWebSocketClientsAsync 找不到任何 connectionId」的問題，後來統一改為 6 小時。

### TTL 不一致風險（原始問題）

若 `SetAddAsync` 每次都刷新 TTL，會出現以下問題：

```
T0: 用戶 A 建立連線
  - websocket:connection:abc123 (TTL 6hr)
  - websocket:liveSession:live001:connections 加入 abc123 (TTL 6hr)

T1 (5.5 小時後): 用戶 B 加入同一直播間
  - websocket:liveSession:live001:connections 的 TTL 被更新為 6hr（因為 SetAddAsync 會刷新）

T2 (6 小時後):
  - websocket:connection:abc123 過期被刪除 ❌
  - websocket:liveSession:live001:connections 仍存在，且包含 abc123 ✅

結果: Set 中存在殭屍 ConnectionId，需要依賴雙重驗證清理
```

> 突然想到：如果兩個人先後連線，最後加入的 liveSession Set TTL 會被刷新，但早加入的 connection 快取可能先過期，這樣 Set 就會存在很久但內容已經不準確。

### 解法：僅首次建立時設定 TTL

新增兩個 Redis 方法：

**`KeyExistsAsync`**
- 底層指令：Redis `EXISTS`
- 用途：檢查 Key 是否存在於 Redis 中
- 回傳值：`true` 表示存在，`false` 表示不存在
- 效能：O(1)

**`ExpireAsync`**
- 底層指令：Redis `EXPIRE`
- 用途：為已存在的 Key 設定過期時間
- 回傳值：`true` 成功，`false` Key 不存在或失敗
- 效能：O(1)

**實作：**

```csharp
//// 檢查 Set 是否已存在
var setExists = await this._redisService.KeyExistsAsync(liveSessionConnectionsKey);

//// 新增成員到 Set（不再自動設定 TTL）
var added = await this._redisService.SetAddAsync(liveSessionConnectionsKey, connectRequestEntity.ConnectionId);

//// 僅在 Set 首次建立時設定 TTL
if (setExists == false && added)
{
    await this._redisService.ExpireAsync(liveSessionConnectionsKey, ConnectionTtl);
    this._logger.LogInformation(
        "直播場次 Set 首次建立，已設定 TTL {TTL} 小時 - LiveSessionId: {LiveSessionId}",
        ConnectionTtl.TotalHours,
        connectRequestEntity.LiveSessionId);
}
```

**修正後的行為：**

```
T0: 用戶 A 連線到直播間 live001
  1. KeyExistsAsync("websocket:liveSession:live001:connections") → false
  2. SetAddAsync("websocket:liveSession:live001:connections", "conn-A") → true
  3. ExpireAsync("websocket:liveSession:live001:connections", 6hr) → true
  ✅ Set TTL = 6hr

T1 (30分鐘後): 用戶 B 連線到同一直播間
  1. KeyExistsAsync("websocket:liveSession:live001:connections") → true
  2. SetAddAsync("websocket:liveSession:live001:connections", "conn-B") → true
  3. ❌ 不呼叫 ExpireAsync，Set TTL 維持原本剩餘的 5.5hr

T6 (6小時後):
  - Set 過期 ❌
  - 所有單一連線快取也應過期 ❌
  - 資料自動清理完成 ✅
```

## 5️⃣ 廣播前的雙層驗證（Lazy Cleanup）

廣播前不直接信任 Set 的內容，逐一驗證每個 connectionId：

1. `SetMembers` 撈出所有 `connectionId`
2. 逐一呼叫 `GetConnectionCacheEntityAsync(connectionId)`
   - 若 Redis 個別 key **不存在** → 視為殭屍連線，立刻 `SetRemove`
   - 若**存在** → 正常推送

此機制（Lazy Cleanup）處理了「Set 有記錄但個別連線快取已過期」的不一致狀態，不需要額外的背景清理 Job。
