# BackgroundService 搬移至獨立排程的分析

## 現況說明

### 目前有兩個 BackgroundService（跑在 API Pod 裡面）

| Service | 執行間隔 | 用途 |
|---------|---------|------|
| `LiveEventPollingService` | **每 10 秒** | 補撈 Webhook 漏掉的留言（備援機制） |
| `LiveStatusPollingService` | **每 15 分鐘** | 偵測直播是否結束（FB 狀態變 VOD → 更新場次為 Ended，終止抽獎活動，記錄最高觀看人數）|

### 現況架構問題

```
Pod A (Web API)                    Pod B (Web API)
├─ HTTP 請求（Webhook、API）        ├─ HTTP 請求（Webhook、API）
├─ LiveEventPollingService ← 每10秒 ├─ LiveEventPollingService ← 每10秒（同時跑！）
└─ LiveStatusPollingService         └─ LiveStatusPollingService（同時跑！）

replicaCount: 2 → 兩個 Pod 都有 BackgroundService
→ 用 Redis 分散式鎖防止重複處理，但每次都在搶鎖（浪費）
```

---

## 搬移好處分析

### ✅ 好處 1：API Pod 資源不再被搶占

**現狀**：
- 直播高峰期大量 Webhook 湧入，API Pod 忙於 HTTP 請求
- 同時 `LiveEventPollingService` 每 10 秒呼叫 Facebook API，消耗 CPU / 記憶體 / 網路
- 競爭資源可能讓 Webhook 回應超過 Facebook 5 秒上限 → **留言掉失**

**搬移後**：
- API Pod 專心處理 HTTP 請求
- Polling 在獨立 Pod 執行，資源完全隔離 → **降低留言掉失風險**

---

### ✅ 好處 2：資源可獨立設定

**現狀**：
- 想增加 Polling 資源只能提升整個 API Pod 的 cpu/memory → 浪費預算

**搬移後**：
```yaml
# API Pod
api:
  resources:
    cpu: "500m"
    memory: "500Mi"

# Polling Worker（可獨立設定）
worker:
  resources:
    cpu: "100m"
    memory: "128Mi"
```
- 各自按需調整，成本更精準

---

### ⚠️ 好處 3：消除 Redis 搶鎖問題

**現狀**：
- 2 個 Pod 同時都想執行 Polling → 每次都要搶 Redis 分散式鎖
- 每 10 秒重複搶鎖 → 浪費 Redis 連線，行為不透明

**搬移後**：
- **CronJob**：天生只有 1 個執行者，完全不需要搶鎖 ✅
- **Deployment**：需保持 `replicas: 1` 且不開 HPA，才能避免搶鎖 ⚠️

---

### ⚠️ 好處 4：AutoScaling 時行為可預測

**現狀**：
- 流量暴增 → HPA 擴展 API Pod → 突然多出一個 BackgroundService
- Redis 鎖介入處理，邏輯複雜、難以預測

**搬移後**：
- **CronJob**：每次只起一個 Pod，完全不受 HPA 影響 ✅
- **Deployment**：需主動確保不設 HPA，否則相同問題重現 ⚠️

---

### ⚠️ 好處 5：部署 API 不中斷 Polling

**現狀**：
- 部署新版 API → Rolling Update → BackgroundService 隨 Pod 重啟而停止
- 若重啟期間 Webhook 也掛掉 → **雙重失效**

**搬移後**：
- 若 Worker 使用**獨立 Docker Image** → 部署 API 完全不影響 Polling ✅
- 若 Worker 使用**相同 Docker Image**（目前是同一個）→ 部署時 Worker 也會重啟，好處只有一半 ⚠️

---

### ⚠️ 好處 6：執行紀錄清晰可查

**現狀**：
- API 請求、Polling 執行、Webhook 處理的 Log 全部混在一起
- 難以查詢「這次 Polling 是否成功」或計算執行時間

**搬移後**：
- **CronJob**：每次觸發建立獨立 Pod，有 per-execution 的成功/失敗狀態 ✅
  ```
  kubectl get pods
  polling-job-1741500000-abc  Completed  0/1
  polling-job-1741500060-xyz  Failed     0/1
  ```
- **Deployment**：Log 仍為連續串流，需靠時間戳記搜尋，沒有 per-execution 狀態 ⚠️

---

## 建議搬移方式

| Service | 建議方式 | 理由 |
|---------|---------|------|
| `LiveEventPollingService`（10 秒）| **獨立 Deployment（Worker Pod）** | K8s CronJob 最小單位為 1 分鐘，10 秒間隔只能用長駐 Worker |
| `LiveStatusPollingService`（15 分鐘）| **Kubernetes CronJob** | 間隔長，天生適合 CronJob，可獲得全部好處 |

## Redis Feature Toggle（目前兩個 Service 都有）

兩個 Service 都透過 Redis Key 控制開關，**預設為關閉**：

| Service | Redis Key |
|---------|-----------|
| `LiveEventPollingService` | `Nine1:Livebuy:Polling:Enabled` |
| `LiveStatusPollingService` | `Nine1:Livebuy:LiveStatusPolling:Enabled` |

搬移後這個開關機制可以保留或替換為 K8s CronJob 的 `suspend: true/false`。
