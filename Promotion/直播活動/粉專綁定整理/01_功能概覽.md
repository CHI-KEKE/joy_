# 粉專綁定功能概覽

## 這個功能是在做什麼

讓「商店後台」可以安全地綁定、管理 Facebook 粉專，作為之後「直播帶貨」的基礎設定。

```
沒有粉專綁定 → 就不能直播
粉專授權壞掉 → 直播功能要擋
粉專被拔權限 → 系統要能知道、提醒、重綁
```

---

## 為什麼需要「粉專綁定」

Facebook 直播帶貨有幾個前提：

- 直播是「粉專」層級，不是個人帳號
- 系統必須拿到粉專的授權 Token
- Facebook 要能主動通知我們（Webhook）留言、直播事件

Facebook 的規則：「你要我把某個粉專的留言事件推給你，那你要先被這個粉專『正式授權』。」
這個授權關係，就是「綁定」。

### 整條用戶旅程

```
商家：綁粉專 → 建活動 → 設商品/關鍵字 → 開播
顧客：留言關鍵字 → 收私訊 → 點連結 → 加車結帳
```

### 系統與 Facebook 之間的互動

| 互動 | 說明 |
|------|------|
| OAuth 授權 | 取得 Token |
| Webhook | 即時收到留言事件 |
| Messenger | 發送私訊 |
| Live API | 控制直播、抓狀態 |

---

## 在後台的實際操作流程

進入路徑：**直播活動站台 → 直播串流設定**

#### 1️⃣ 第一次進來（未登入 Facebook）

畫面顯示「尚未登入 Facebook」與「登入 Facebook」按鈕，點擊後進入 Facebook OAuth 流程。

#### 2️⃣ 登入 Facebook（OAuth 流程）

- 系統另開分頁導向 Facebook 登入頁
- 使用者登入並同意授權
- Facebook 導回後端 API：`/api/facebook/oauth/callback`
- 後端處理完畢後，自動關閉分頁並導回設定頁刷新

> 前端完全不碰 App Secret。

#### 3️⃣ 新增粉專

登入成功後顯示「新增粉絲專頁」按鈕與已啟用粉專清單。

點「新增粉絲專頁」時：
- **情況 A**：Facebook 帳號底下沒有粉專 → 彈窗「粉絲專頁無法連接」
- **情況 B**：有粉專 → 顯示粉專清單彈窗（已啟用的不可勾選，未啟用的可多選）

#### 4️⃣ 啟用粉專

- 勾選粉專（最多 10 個）
- 點「連結」→ 呼叫後端 API
- 成功後關閉彈窗、頁面刷新、顯示最新啟用清單

#### 5️⃣ 解除綁定 / 重新連結

- **解除綁定**：跳確認視窗，後端標記為「Disconnected」（不是硬刪）
- **重新連結**：重新走一次 Facebook OAuth

---

## 工作分配

前端只負責 UI 與導頁，所有 Facebook 授權、Token、綁定邏輯全部在後端完成。

---

## 架構

| 元件 | 用途 |
|------|------|
| Facebook Graph API | 取得 Token、粉專資料、訂閱 Webhook |
| AWS Secrets Manager | 儲存加密 Token |
| DynamoDB | 儲存綁定記錄 |
| Redis | 快取 |

---

## 「綁定」代表的三件事

### ① 身分授權（Identity）

系統可以「代表這個粉專」呼叫 Facebook API。  
技術上是取得 **Page Access Token**，這個 Token 就是粉專的「身分證」。沒有它，什麼都不能做。

### ② 權限授予（Permission）

OAuth 時一次同意的能力：
- 讀留言
- 管理直播
- 發佈內容
- 管理粉專資料

### ③ 技術通道（Event Channel）

Facebook 會主動推事件（Webhook）：
- 有人留言
- 直播開始
- 直播結束

---

## 一筆綁定記錄所代表的內容

| 欄位 | 說明 |
|------|------|
| 粉專 ID | 你是誰 |
| 商店 ID | 是哪一間店 |
| Page Token 參照 | 怎麼代表你 |
| Token 到期時間 | 什麼時候要重授權 |
| Webhook 是否成功 | 事件有沒有接上 |
| 綁定狀態 | 現在還能不能用 |

### DynamoDB 資料關係

- 一店可綁多粉專
- 一粉專可被多店綁定
