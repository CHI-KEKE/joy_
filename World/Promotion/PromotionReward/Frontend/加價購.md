
## 概念

「加價購」是指顧客在購買特定商品或達到一定門檻後，可以用比原價更便宜的價格額外購買特定商品，它不是免費贈品、也不是直接折扣，而是一種「額外購買權限（Right to Buy）」

這個活動的本質是「主商品帶動附屬商品的銷售」。商業上是提高客單價（AOV, Average Order Value）的一種手法。



## 加價購的資格要根據「折扣後金額」來判斷

加價購的門檻通常是「滿 $X」或「買指定商品」。而這個「滿 $X」通常是指折扣後的小計金額（即顧客實際購買的商品價值），所以要等商品層活動（如任選、第 N 件折扣、現折）、折價券、優惠碼，這些都跑完後，金額才是穩定的，此時再判斷是否達加價購門檻


## 要在抵價券之後，點數折抵之前

💡 為什麼在「抵價券之後」？

抵價券會直接降低購物車金額。
所以活動通常會定義為：「加價購門檻不含抵用券後金額」。
也就是說：

折價券折完後的小計還要滿 $1000，才能加購。

👉 如果你把加價購放在抵價券之前，就會錯誤地用「折前金額」判門檻。
導致顧客折完後實際金額低於門檻但仍能加購，造成條件錯誤

💡 為什麼在「點數折抵之前」？

點數折抵是顧客的支付行為，屬於「付款階段」的動作。但加價購是「購物階段」的活動（加入購物車時判定）。你必須在顧客抵點數前，先確定他是否有加購資格。
否則點數折完後金額會下降，會出現「資格閃爍」的問題（前端 UI 顯示加購→再被移除）

## 避免重複判斷與贈品干擾

加價購會加入新商品（例如袋子、旅行組），這些「被加購的商品」如果放得太早，系統會誤把它們算入其他活動的件數或金額，造成錯誤。

舉例：

「滿三件送贈品」活動

顧客因加價購又多一件商品 → 系統若重算件數，就可能誤以為「滿四件」再多送一份贈品。所以必須放在滿件贈之後，確保加購品不影響前面活動的門檻

## 排除邏輯實作

**RuleInitProcess**

<br>

會根據不同活動類型實作
```csharp
/// <summary>
/// The RuleInitProcess
/// </summary>
public override void RuleInitProcess()
{
    this.ExclusiveTags ??= new HashSet<string>();
    this.ExclusiveTags.Add(FlagConstants.AddOnsSalepageSub);
    this.ExclusiveTags.Add(FlagConstants.CouponExcludedByOrder);
}
```

<br>

促購前台有一步 LoadRules，每個活動 Init 流程實作需要排除哪些 Flag 的商品，接著計算時 PurchasedItems 會需要把該 Tags vs Flag 交集有結果要排除掉

<br>

```csharp
/// <summary>
/// Gets the PurchasedItems
/// </summary>
public IEnumerable<PurchasedItem> PurchasedItems =>
    this._currentRule == null ?
        this._purchasedItems :
        this._purchasedItems.Where(x => ((this._currentRule.ExclusiveTags ?? new HashSet<string>()).Union(this.ExclusiveTags))
                                       .Intersect(x.Flags).Any() == false
                                        && (this._currentRule.GlobalExcludedProductIds == null ? true : this._currentRule.GlobalExcludedProductIds.Contains(x.Product.Id) == false)
                                   );
```
