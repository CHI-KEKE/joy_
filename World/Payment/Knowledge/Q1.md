## 1. 發卡行 (Issuer)

就是把卡片「發給消費者」的銀行。

<br>

例如你在 台新銀行 申辦一張 Visa 信用卡，那台新就是你的 發卡行。

<br>

當你刷卡時，它要檢查你「有沒有額度、卡片狀態正不正常」 決定要不要批准。

<br>

如果你後續要退款，還是得透過它來把款退回來。

<br>

---

## 2. 收單行 (Acquirer)

幫商家（店家、電商平台）處理信用卡交易的銀行。例如：你在某電商刷卡買東西，那個電商的收單行可能是 中國信託。

<br>

授權：收到商家的交易資料  送去卡組織  再轉給發卡行  問「這筆交易能不能過？」

<br>

清算/結算：事後幫商家把錢收回來，再扣掉一些手續費後，把錢打給商家。

<br>

---

## 3. 卡組織 (Card Network)

Visa、MasterCard、JCB、American Express

<br>

它不是銀行，而是「規則制定者 + 網路提供者」。

<br>

制定規則：規定授權流程、錯誤碼格式、手續費怎麼算。

<br>

提供網路：建一個安全快速的「高速公路」，讓收單行的請求能送到發卡行。

<br>

保障消費者：例如「爭議款仲裁」，你覺得刷卡被盜刷，Visa 會有規則讓你可以申訴。

<br>

---

## 4. 整體流程（簡化）

你在商店刷卡  商家系統把交易請求送到金流服務（例如 Stripe、Razer、2C2P ) 接著把交易資料交給 收單行

<br>

#### 授權 (Authorization)

收單行  卡組織  發卡行，確認這張卡是否有效、餘額是否足夠。

<br>

發卡行回覆「核准」或「拒絕」。

<br>

這時候錢還沒到商家，只是「先鎖住持卡人的額度」。

<br>

#### 清算 (Clearing) & 結算 (Settlement)

每天收單行會把所有交易彙總，透過卡組織發給發卡行。

<br>

發卡行把錢撥給卡組織，卡組織再到收單行，最後收單行把款項給商家。

<br>

同時各方（發卡行、收單行、卡組織）會抽手續費。

<br>

#### 商家入帳

依不同金流供應商（Stripe 可能 T+2、T+7），錢才會真的進商家帳戶。

<br>

 「授權成功後我們會立刻更新訂單狀態，但清算/結算是延遲的，所以要設計 reconciliation（對帳）機制，避免商家入帳金額和系統記錄不一致。」

<br>

## 5. 「誰」來決定是否可以分期

是否允許分期，主要是由發卡行決定，因為分期本質上就是「發卡行願意把這筆款項攤成多期貸款給持卡人」。

<br>

---

## 6. 商家本身能不能限制分期？

比如只開放某些銀行的分期？

<br>

發卡行 → 負責核准分期、計算利率。
商家 → 在前端可以設定「支援哪些銀行的分期方案」。
收單行 / 金流服務 → 提供 API，回傳支援的 bank list。

<br>

我們的資料結構裡會有 installmentType、bankList、hasInterest，前端就能正確渲染分期選項

<br>

## 7. 信用卡號輸入後查詢發卡行

信用卡號輸入後，我們通常想知道「發卡行」是哪一家，這要怎麼查

<br>

輸入信用卡號後，前 6～8 碼（BIN code / IIN, Issuer Identification Number） 就能判斷出：

<br>

卡別（Visa、MasterCard、JCB…）
發卡行（例如台新、國泰世華、HSBC、AmBank…）
卡片等級（白金、商務…有些 BIN 表可以分）
國別

<br>

#### 實作方式

維護一份內部的 BIN Code 對照表（常見做法，放資料庫）。
使用第三方金流的查詢 API（例如 Stripe、Razer，會回傳 issuer、country 等欄位）。
購買外部 BIN Database（有些公司專門提供最新 BIN 資料）。

<br>

## 8. 刷卡失敗常見原因？串接要注意什麼？

### 8.1 使用者端原因

| 原因          | 說明            | 處理方式                      |
| ----------- | ------------- | ------------------------- |
| 卡號輸入錯誤      | 使用者手動輸入錯誤     | 前端檢核格式（Luhn 演算法），回饋「卡號錯誤」 |
| 信用卡過期       | 卡片有效期已過       | 提醒使用者更新卡片                 |
| CVV / 安全碼錯誤 | CVV 錯誤        | 提醒使用者重新輸入                 |
| 卡片限額不足      | 單筆交易超過信用卡可用額度 | 回饋「餘額不足或超額」               |
| 卡片不支援該支付方式  | 例如海外卡不支援分期    | 前端篩選或提示                   |
| 多次嘗試失敗被鎖卡   | 銀行端安全措施       | 提示使用者聯絡發卡行                |

<br>

### 8.2 金流 / PSP / 銀行端原因

| 原因                       | 說明                         | 處理方式                       |
| ------------------------ | -------------------------- | -------------------------- |
| 授權失敗                     | 發卡行拒絕授權                    | 解析錯誤碼 → 給使用者明確提示，如「銀行拒絕授權」 |
| 銀行風控拒絕                   | 高風險交易、海外交易或異常行為            | 提示「銀行拒絕」，建議使用者聯絡銀行         |
| PSP 錯誤 / 連線問題            | 服務短暫不可用或 Timeout           | 可做重試，或回饋「系統忙，請稍後再試」        |
| 幣別或國別限制                  | 發卡行不支援該幣別                  | 交易前檢查幣別與卡別                 |
| 卡組織拒絕                    | Visa / MasterCard / JCB 拒絕 | 解析組織錯誤碼，回饋給使用者             |
| 分期限制                     | 卡片不支援分期或分期期數不合法            | 前端篩選支援卡片與分期方案              |
| BIN 不匹配 / ChannelCode 錯誤 | PSP 需要正確帶上 ChannelCode     | 交易前依 BIN 對照表查詢，確保帶正確參數     |

<br>

### 8.3 系統端 / 技術原因

| 原因                     | 說明                 | 處理方式                        |
| ---------------------- | ------------------ | --------------------------- |
| Timeout                | 與 PSP / 銀行連線超時     | 設定合理 Timeout，必要時重試一次，避免重複扣款 |
| 資料格式錯誤                 | API 請求缺少必填欄位或格式不正確 | 交易前做資料檢核、日誌記錄錯誤             |
| 金額不一致                  | 前端傳送金額與伺服端計算不一致    | 對交易做簽名 / 檢查                 |
| 重複交易                   | 客戶或系統重複提交          | 做 Idempotency（唯一交易號）防止重複扣款  |
| Callback / Webhook 沒收到 | PSP 回傳結果遺失或延遲      | 定期輪詢查詢訂單狀態，避免錯誤判斷           |

<br>

### 8.4 特殊情況

| 原因             | 說明            | 處理方式                   |
| -------------- | ------------- | ---------------------- |
| 部分授權 / 部分失敗    | 發卡行允許部分金額     | 前端 / 後端需處理差額或補差額流程     |
| 信用卡爭議 / 退款問題   | 使用者先發起退款或爭議   | 系統需能同步更新狀態，避免重複扣款或錯誤狀態 |
| 3D Secure 驗證失敗 | 使用者未完成 3DS 驗證 | 提示完成驗證或取消交易            |

<br>

## 9. 金流整合時，你覺得最常遇到的技術問題有哪些

### 9.1 錯誤碼不一致 時與對方溝通的窗口

現象：文件寫「E1001 = 金額錯誤」，但實際 API 回來卻可能是 10001 或乾脆沒有定義，導致難以對應。

<br>

本質問題：每個 PSP（Payment Service Provider）像是 Stripe、2C2P、Razer 都有自己的錯誤碼系統，但我們系統要能「解讀並統一處理」。

<br>

### 9.2 Timeout vs 成功入帳

現象：你送出交易，結果 API Timeout，但銀行可能已經扣款成功。

<br>

本質問題：網路斷線 ≠ 交易失敗。必須有「補單」(Retry) 或「對帳」(Reconcile) 機制。

<br>

### 9.3 重複交易

現象：使用者狂按「付款」，結果扣了兩次。

<br>

解決方案：要有「冪等機制」（Idempotency），或用「唯一交易流水號」。

<br>

### 9.4 簽章 / 加密差異

現象：每個 PSP 都要求不同演算法：有的用 HMAC-SHA256，有的用 RSA，有的還要 Base64。

<br>

本質問題：這是「安全驗證」問題，必須依 PSP 規格正確生成簽章，否則交易會被拒。

<br>

### 9.5 幣別 / 金額對應

現象：消費者付 TWD，但 PSP API 要 USD；有的用 1000.00，有的用 100000 cents。

<br>

本質問題：必須有「統一的金額換算邏輯」，包含 匯率處理 與 小數點 rounding。

<br>

### 9.6 Callback / Notification 不一致

現象：前端顯示「付款失敗」，但後端 Callback 卻說「成功」。

<br>

本質問題：「使用者看到的畫面 ≠ 最終真相」，要以 Callback 為最終依據，確保訂單狀態正確。

<br>

### 9.7 API 格式差異

現象：有的用 JSON，有的用 XML，有的用 Form-UrlEncoded。

<br>

解法：建立一層 middleware，把內部統一格式轉成各 PSP 要的格式。

<br>

### 9.8 測試環境 vs 正式環境

現象：Sandbox 永遠成功，正式上線後才遇到「銀行維護」、「風控拒絕」。關帳天數實際難以測試

<br>

本質問題：測試環境不等於真實世界，需要有「異常狀況模擬機制」。

<br>

## 10. 客人付款成功，但我們後端的訂單狀態沒有更新成功，你會怎麼設計系統來避免或修正這種情況？

### 10.1 系統層面

DB 寫入失敗（Transaction rollback、連線 timeout）
程式碼 exception（序列化錯誤、Mapping 問題）
Callback 收到但處理失敗

<br>

### 10.2 流程層面

只依賴前端「同步回應」更新訂單，但真正準確的是「後端 callback」。
Callback 延遲或 PSP 端漏發。

<br>

### 10.3 業務層面

付款成功，但金額、訂單狀態不符合業務規則 → 被系統拒收。

<br>

### 10.4 即時層（以 Callback 為準）

訂單狀態更新邏輯只接受 PSP callback 作為最終依據。

<br>

冪等性處理 即使 callback 來 3 次，也只能更新一次，不會重複扣款或覆蓋正確狀態。

<br>

錯誤重試機制 如果 callback 寫 DB 失敗，事件要丟到 MQ（Message Queue） 或 Dead Letter Queue，稍後再重試。

<br>

### 10.5 補償層（定期對帳 Job）

定時對帳：每天或每小時，比對「我們的訂單」與「PSP 訂單」。
補單機制：若發現 PSP 端顯示「已成功」，但我們系統顯示「未付款」，就要自動補單。

<br>
