## 1. 次特店

如果你是開一家小網店，你不會自己跑去銀行申請這個特店編號（流程太麻煩、門檻高）。

<br>

所以你會透過像「91APP」這樣的平台來開店，刷卡收款就走 91APP 的金流。

<br>

你不是直接跟銀行簽約的商家

<br>

你是「掛在 91APP 底下」的小商店

<br>

這樣的商店，就叫做：

<br>

🧾 次特店（Sub-Merchant）

<br>

而91APP就是：

<br>

🏢 主特店 / 特店平台 / 金流平台（Merchant Aggregator）

<br>

銀行只認得平台的「特店編號」，平台內部用「次特店 ID」來管理不同商店，平台是收款總管，商家只是借用這個通道來收款

<br>

對銀行來說，次特店制度有什麼風險？→ 銀行要靠平台來做 KYC（身分驗證）與風控，若平台沒做好，會導致詐欺、洗錢等風險

<br>

## 2. NCCC

NCCC = 聯合信用卡處理中心（National Credit Card Center）

<br>

簡單說，它是台灣的信用卡清算與收單中樞機構之一。你可以把它想成，「信用卡交易的高速公路管理局」所有交易都可能要走它的路，才會到銀行和商店手上

<br>

91APP 是使用 NCCC 提供的收單服務來處理某些支付方式（像 Apple Pay、Google Pay），而不是說 91APP「只」跟 NCCC 簽約，或「沒有跟銀行簽約」。像 Apple Pay 和 Google Pay 這類支付方式，在目前台灣的金流架構下，可能只能透過 NCCC 處理（也就是說：Apple Pay 的交易只能走 NCCC 這條路）

<br>

## 3. TapPay

TapPay（支付通） 是一家台灣的 第三方金流服務商 / 金流平台

<br>

幫助電商、APP、系統開發者快速整合收款功能（像是信用卡、Apple Pay、LINE Pay 等），不用自己跑去跟銀行簽一堆約。

<br>

TapPay 是民營的金流服務商，整合多種收單銀行與支付工具；NCCC 是台灣國家級的清算平台，負責處理卡組織交易。

<br>

## 4. 根據 App 條件來選擇正確的 PSP（像 TapPay）

可以判斷 app 版號 & 商店設定決定該筆交易的 Apple Pay 要走哪一個 PSP，並且

<br>

載入正確的 PSP 的 SDK，如為 PSP=TapPay，再查詢收單行設定，並至信用卡子代碼查詢取 proxyMID 對正確的 PSP 送出交易請求

<br>

功能漸進部署（Feature Flag by version），可能在新版 App 中才支援某個 PSP（例如加入 TapPay 支援），所以舊版 App 要走舊的 PSP、新版才能走新的 SDK

<br>

## 5. 金流 SDK

| 類別               | 內容                          | 說明                       |
| ---------------- | --------------------------- | ------------------------ |
| 🔐 加密與安全         | 金融級加密模組（如 RSA/AES）          | 確保用戶卡號與交易資料不外洩           |
| 💳 信用卡輸入 UI      | 可客製化的卡號、到期日、CVC 輸入欄位        | 符合 PCI-DSS 安全規範          |
| 📱 行動支付整合        | Apple Pay / Google Pay 授權流程 | 包含 wallet 授權、token 拿取    |
| 📦 Token 產生流程    | 將卡號等資料換成 token              | 提交交易前，不讓商家接觸卡號本身         |
| 📡 API 封裝        | 封裝好的交易送出方法                  | 如 `sendPaymentRequest()` |
| 🎯 交易事件回傳        | 成功、失敗、取消等 callback 機制       | 方便開發者處理 UI 與錯誤           |
| 📊 錯誤代碼 & 訊息處理   | 回傳錯誤代碼與對應錯誤文字               | 幫助 debug 與 UX 處理         |
| 🛡️ 3D Secure 支援 | 符合國際卡組織的安全驗證流程              | 如彈跳 OTP、瀏覽器驗證            |
| ⚙️ SDK 初始化邏輯     | 設定 merchantId、APIKey 等參數    | 初始化時設置商店身分               |
| 🌍 支援語系          | 英文 / 繁中 / 日文等               | 多語系界面與錯誤訊息               |
| 🔧 Log 記錄（開發用）   | Debug 模式下輸出內部狀態             | 不可在正式機開啟                 |

## 6. 3D 驗證

Three Domain Secure（三方域驗證）

<br>

它是一種由國際卡組織（Visa、MasterCard、JCB 等）制定的線上信用卡付款「身份驗證機制」，

<br>

目的是：確認這張卡的使用人真的是持卡人本人。

<br>

線上刷卡的「第二道門檻」 → 雙重驗證（2FA）的金流版本

<br>

| Domain（領域）  | 指的是誰                      |
| ----------- | ------------------------- |
| 1️⃣ 發卡機構領域  | 持卡人所屬的發卡銀行（如中信）           |
| 2️⃣ 收單機構領域  | 商家端的收單銀行（如台新）             |
| 3️⃣ 互聯網支付領域 | 中間的驗證與清算網路（如 Visa/Master） |

<br>

## 7. BIN Code 判斷外國卡

| 卡號開頭    | 代表含義         |
| ------- | ------------ |
| 4000 01 | Visa，某台灣發卡銀行 |
| 6011 11 | Discover，美國卡 |
| 3566 66 | JCB，日本發卡行    |

<br>

因為每個國家的發卡銀行都有獨特的一段 BIN 編號範圍，所以你只要查這幾碼，就可以知道是不是「境外卡」。

<br>

例如：

<br>

台灣發卡的 Visa 卡通常開頭是 4 開頭、特定範圍

<br>

美國發卡的 Discover 開頭可能是 6011

<br>

新加坡發卡的 MasterCard 可能是 5213 開頭

<br>

## 8. WebView

WebView 是 App 內嵌的一個「網頁容器」，可以在 App 裡打開並顯示網頁內容（HTML / JS / CSS），但不跳出 App。你可以把 WebView 想成 App 中的一個「迷你瀏覽器」。它不是 Safari、Chrome，而是 App 裡內建的一個畫面元件，用來載入網站內容。

<br>

| 用途           | 說明                         |
| ------------ | -------------------------- |
| 顯示金流頁面       | 像付款完成畫面、訂單明細頁              |
| 顯示第三方協力平台    | 像 LINE Pay、Apple Pay 回傳頁   |
| 嵌入後台、問卷、客服頁面 | 直接使用既有網站畫面                 |
| 跨平台開發        | 一套網頁讓 Android/iOS 都能顯示相同介面 |

<br>

## 9. WebView 和 App 要怎樣有良好溝通

這是整個串接 Apple Pay/TapPay 成功與否的關鍵。

<br>

因為兩者是不同層級：

<br>

App 是原生層（Native）

<br>

WebView 是網頁層（HTML+JS）

<br>

所以溝通要靠「橋接技術（Bridge）」來串起兩邊。

<br>

| 方向            | 橋接技術                                  | 用途                      |
| ------------- | ------------------------------------- | ----------------------- |
| WebView → App | JS → Native Bridge（JavaScript 調用原生函式） | 網頁告訴 App：我要啟動 Apple Pay |
| App → WebView | WebView.evaluateJavascript()          | App 執行 JS：告訴網頁交易完成      |

<br>

## 10. 非同步事件都有對應 callback 是指什麼？

「非同步事件」就是一種：你發出請求後，結果不是立刻回來，而是『稍後透過 callback 回來』的事件

<br>

請求 Apple Pay 開始授權

<br>

TapPay SDK 要去跟 Apple、發卡銀行、收單銀行對接

<br>

整個流程可能需要幾秒，甚至十幾秒（尤其在網路差的狀況）

<br>

➡️ 如果這時候你是「同步等待」→ App 卡死、UI 卡住、用戶覺得當掉了

<br>

➡️ 所以「非同步 + callback」是標準做法

<br>

## 11. Apple Pay 的憑證

Apple Pay 憑證（Apple Pay Merchant Certificate），是由 Apple 核發給「商店（商家）」的一個「身分認證」數位憑證。一份讓你的 App（或網站）可以合法呼叫 Apple Pay 並取得付款 token 的「身份證 + 安全通行證」

<br>

🍎 Apple Pay 的核心原則是：

<br>

讓商家永遠無法碰到消費者卡號 → 透過 token + 加密 + 憑證，完全封裝金流路徑

<br>

Apple 不相信你（也不信 PSP）會做好風控，所以整個流程設計成：

<br>

Apple 保管卡號

<br>

Apple Pay Token 是一次性、短效的

<br>

憑證加密 token → 只能給特定 PSP 解開

<br>

憑證就是用來「授權解密」的唯一身分識別

<br>
