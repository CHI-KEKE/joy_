# 商品加價購 加購頁 — 完整流程與篩選/檢核機制現狀

> Story #582440 · AC 檢核位置（商品加價購加購頁）
>
> 範例連結：`https://10230.shop.qa.91dev.tw/V2/AddOnsSalePage/ExtraPurchaseList/980491?shopId=10230`

---

## 一、位置定義

| 項目 | 說明 |
|------|------|
| **頁面名稱** | 商品加價購 加購專區 |
| **URL 格式** | `/V2/AddOnsSalePage/ExtraPurchaseList/{mainSalePageIdOrCode}?shopId={shopId}` |
| **路由參數** | `{id}` = **主商品**的 salePageId 或 salePageCode（非 promotionId） |
| **前端應用** | `ClientApp`（非 CartClientApp） |
| **Controller** | `AddOnsSalePageController.ExtraPurchaseList()` |
| **差異** | 與滿額加價購（`CartExtraPurchase/ExtraPurchaseList/{promotionId}`）不同，這裡帶的是主商品 ID |

## 二、完整呼叫鏈

```BASH
[使用者開啟商品加價購加購頁]
  ↓
AddOnsSalePageController.ExtraPurchaseList(id)
  → ViewBag.MainSalePageIdOrCode = id
  → [RequireReactThemeAttribute] → 返回 ClientApp React View
  ↓
[ClientApp 啟動]
AddOnsSalePageExtraPurchaseListProvider.Init()
  ↓
Step 1: getSalePageV2Model(shopId, 0, mainSalePageIdOrCode)
  → 取得主商品資訊 (SalePageV2 API)
  → 取得主商品的真實 salePageId、IsSalePageBundle、SaleProductSKUIdList
  ↓
Step 2: 同時呼叫（Promise.all）
  [2a] getSalepageExtraPurchase(shopId, salePageId)
       → FTS (SPL): {FtsApiDomain}/salepage-listing/api/salepage/add-ons/{shopId}/{salePageId}
           ↓ SalepageExtraPurchaseService.GetSalepageSearchDataAsync()（SPL 服務層篩選在這裡）
           → 回傳 SalepageExtraPurchaseEntity（含活動設定 + 加購品清單）
  [2b] getSellingQtyListNewModel(skuIdList)
       → 取得主商品庫存
  [2c] getBundleInfoData(shopId, salePageId)  ← 僅當 IsSalePageBundle === true 時呼叫
       → 取得組合商品資訊
  ↓
sortAddOnsSalepageResponse(salepageExtraPurchaseResponse)
  → 依 itemInfo.order 排序加購品（無業務篩選）
  → setAddOnsSalePageList(sortedList)
  ↓
[如果 from=cart]:
  initBasketFromCart()
  → Shopping API: carts/salepage-add-ons?cartUniqueKey={key}&skuId={id}&...
  → 取得購物車中已加購的數量，初始化菜籃狀態
  ↓
AddOnsProductList.tsx
  → addOnsSalePageList.map(salePage => <ThemeCoreProductCard ... />)
  → 直接渲染所有商品，無業務篩選
```

## 三、SPL 服務層的篩選邏輯（`GetSalepageSearchDataAsync`）

這是商品加價購真正的篩選層，位於 SPL Repo（`SalepageExtraPurchaseService.cs`）。

### 3.1 資料來源

SPL 呼叫 SPS 取得加購品商品頁資料（`SalepageListingResponseDataSalepageModel`，不同於滿額加價購用的 `GetSalepageListByIdsResponse`），此型別**包含 `IsClosed`、`SellingStartDateTime`、`SellingEndDateTime`、`IsInvisible`** 等欄位。

```csharp
// 兩種模式（依 isEnabledAddOnsPayDelivery 開關切換）
// Mode A（啟用付款配送加購）: 呼叫 SearchForPresentationAsync，一次批次查詢
//   附加篩選：溫層一致 + 配送方式有交集
// Mode B（一般模式）: 每個加購品呼叫 GetSalepageAsync（逐一查詢）
```

### 3.2 篩選條件（foreach salepage 迴圈內）

```csharp
// 篩選 1：商品頁關閉、非開賣時間 → 排除
if (salepageInfo == null
    || salepageInfo.IsClosed == true
    || salepageInfo.SellingStartDateTime > DateTime.Now
    || salepageInfo.SellingEndDateTime < DateTime.Now)
{
    continue;
}

// 篩選 2：隱賣商品（依商店開關）
if (isEnableHiddenAddOnsSalePage == false && salepageInfo.IsInvisible)
{
    continue;
}

// 篩選 3：至少一個加購品 SKU 狀態為開啟（IsShow = true）
if (matchedAddOnsSkuProducts.Any() == false)
{
    continue;
}
```

### 3.3 注意：`IsShareToBuy` 的處理

```csharp
requestEntity.Filters.IsIncludingShareToBuy = true;  // 故意包含分享活動商品
// 但後續無後置篩選 → 分享活動商品仍會出現在清單
```

## 五、篩選/檢核機制完整清單

### ✅ 有實作的篩選與檢核

| 層級 | 篩選/檢核項目 | 實作位置 | 行為 |
|------|------------|---------|------|
| SPL 服務層 | **商品頁關閉排除** (`IsClosed`) | `SalepageExtraPurchaseService.GetSalepageSearchDataAsync()` | `IsClosed == true` → 排除 |
| SPL 服務層 | **非販賣時間排除** | `GetSalepageSearchDataAsync()` | `SellingStartDateTime > Now` 或 `SellingEndDateTime < Now` → 排除 |
| SPL 服務層 | **隱賣商品控制** (`IsInvisible`) | `GetSalepageSearchDataAsync()` | 依商店開關 `isEnableHiddenAddOnsSalePage` 決定是否排除 |
| SPL 服務層 | **SKU IsShow 檢查** | `GetSalepageSearchDataAsync()` | 至少一個對應的加購品 SKU `IsShow = true`，否則排除 |
| SPL 服務層（Mode A）| **溫層一致** | `GetSalepageSearchDataAsync()` | `Temperature == majorSalepage.Temperature` |
| SPL 服務層（Mode A）| **配送方式交集** | `GetSalepageSearchDataAsync()` | `DeliveryTypeIdsList` 需有交集 |
| 前端顯示 | **售完遮罩** | `AddOnsProductList.tsx` | `isSoldOut` 顯示遮罩，不從清單移除 |
| 前端顯示 | **選購按鈕停用** | `AddOnsProductList.tsx` | `isSoldOut` 時停用按鈕（到貨通知型除外） |
| 加購數量 | **總限購數量** | `addOnsValidation.utility.ts` | `totalQty > addOnsQtyLimit × mainSalePageQty` |

### ❌ 未實作的篩選（缺口）

| 篩選項目 | 原因 | 影響 |
|---------|------|------|
| **組合商品排除** (`IsSalePageBundle`) | SPL `GetSalepageSearchDataAsync` 未排除（`SalePageModes` 可取得但未使用） | 組合商品可能出現在加購清單 |
| **分享活動商品排除** (`IsShareToBuy`) | SPL 查詢時 `IsIncludingShareToBuy = true`（故意帶入），且後置未篩除 | 分享活動商品可能出現 |

## 六、與滿額加價購加購頁的關鍵差異

| 比較項 | 商品加價購加購頁 | 滿額加價購加購頁（ViewOnly / from=cart） |
|--------|---------------|----------------------------------------|
| **Controller** | `AddOnsSalePageController` | `CartExtraPurchaseController` |
| **URL id 參數** | 主商品 salePageId / Code | 活動 promotionId |
| **SPL 資料來源型別** | `SalepageListingResponseDataSalepageModel`（SPS 直接搜尋） | `GetSalepageListByIdsResponse`（批次 by ID 查詢） |
| **`IsClosed` 排除** | ✅ **SPL 服務層排除** | ❌ `GetSalepageListByIdsResponse` **無此欄位**，未排除 |
| **販賣時間檢查** | ✅ **SPL 服務層排除** | ❌ `GetSalepageListByIdsResponse` 有欄位但**未使用** |
| **SKU IsShow 檢查** | ✅ SPL 服務層排除 | ❌ 未實作 |
| **隱賣商品控制** | ✅ 依商店開關 | ❌ `includeInvisibleSalepage: true`（一律帶入），未後置篩選 |
| **`isSoldOut` 處理** | FTS 回傳，前端顯示遮罩（不移除） | FTS `BuildPromotionEntitiesAsync` 直接排除（不回傳） |
| **組合商品排除** | ❌ 有欄位但未篩選 | ❌ 有欄位但未篩選 |
| **分享活動排除** | ❌ 故意帶入未篩選 | ❌ 故意帶入未篩選 |
| **加購數量限制** | ✅ `qtyLimit × mainSalePageQty` 檢核 | ❌ 無此邏輯 |

## 七、重要結論

1. **商品加價購在 SPL 服務層有完整的商品狀態篩選**：`IsClosed`、販賣時間、SKU IsShow、隱賣開關，這些在滿額加價購中**完全缺失**
2. **兩者共同的篩選缺口**：組合商品（`IsSalePageBundle`）、分享活動商品（`IsShareToBuy`）— 兩個 endpoint 均未排除
3. **資料型別差異是根本原因**：商品加價購用 SPS 搜尋（`SalepageListingResponseDataSalepageModel` 有 `IsClosed`），滿額加價購用 by-ID 批次查詢（`GetSalepageListByIdsResponse` 無 `IsClosed`）
4. **`isSoldOut` 行為不同**：商品加價購前端顯示遮罩（保留在清單）；滿額加價購 SPL 服務層直接排除
