## 一、位置定義

| 項目 | 說明 |
|------|------|
| **AC 名稱** | 購物車頁 P1 下方的活動加購區區塊 |
| **QE 標記** | `data-qe-id="body-reach-price-extra-purchase-container"` |
| **入口** | 使用者進入購物車頁（`/ShoppingCart/Index?shopId={shopId}`） |
| **前端應用** | `CartClientApp`（獨立於 ClientApp，購物車專屬 React App） |
| **特點** | P1 加購區**有直接顯示商品卡片**，非只有連結 |

## 二、完整呼叫鏈

```bash
[使用者進入購物車頁]
  ↓
ShoppingCartController.Index()
  → RequireReactTheme → 返回 CartClientApp View
  ↓
[CartClientApp 啟動 XState 機器]
  cart.machine.ts
  ↓
getCartReachPriceExtraPurchase 狀態
  → cond: !context.cartReachPriceExtraPurchaseList
  → 呼叫 service: fetchCartReachPriceExtraPurchase
  ↓
cart.service.ts: getCartExtraPurchaseList(cartUniqueKey)
  → HTTP GET: /shopping/api/carts/cart-extra-purchase?cartUniqueKey={key}&shopId={shopId}
  → 目標: Shopping API（Cart Domain，不在本 repo）
      ↓ Cart Domain 判斷購物車符合哪些 CartReachPriceExtraPurchase 活動
      ↓ 對每個符合活動，**內部呼叫** FTS (SPL):
      → FTS: salepage-listing/api/salepage/cart-extra-purchase/{shopId}/{promotionId}
          ↓ BuildPromotionEntitiesAsync()  ← 商品篩選發生在這裡
          → 排除 IsSoldOut（僅此一項）
          → IsSalePageBundle / IsShareToBuy / IsClosed 均未排除
      ↓ 加上 isAppOnly 等 Cart Domain 層資訊後回傳
      ※ 強推斷依據：兩個 API 的 ICartExtraPurchaseSalePage 型別結構完全一致
  ↓
onDone → action: updateCartReachPriceExtraPurchase
  → assign: cartReachPriceExtraPurchaseList = event.data || []
  (原始資料直接存入，無任何篩選)
  ↓
CartContext (cart.provider.tsx)
  → 傳遞 cartReachPriceExtraPurchaseList 給子元件
  ↓
CartReachPriceExtraPurchaseBlock (cartReachPriceExtraPurchaseBlock.tsx)
  ↓
  [篩選 1] filter(!item.isAppOnly)   ← Web 版排除 App 限定活動
  [篩選 2] displayList.slice(0, 5)   ← 最多顯示 5 檔活動
  ↓
  for each extraPurchase:
    ReachPriceExtraPurchaseCard (reachPriceExtraPurchaseCard.tsx)
      ↓
      convertCartExtraPurchaseSalePageListToEntity(salePageList)
        [僅做格式轉換，無業務邏輯篩選]
        ※ salePageModes → DEFAULT_SALEPAGE_MODES = ''（API 未回傳此欄位）
        ※ isSoldOut 欄位在轉換時丟失（未映射到 IPurchaseExtraSalePageEntity）
        ※ sellingStartDateTime/listingStartDateTime → null（API 未回傳）
      ↓
      [條件渲染] if (entityList.length === 0) return null
      ↓
      SalepageList (salepageList.tsx)
        [篩選 3] data.slice(0, MAX_SALEPAGE_COUNT=100)  ← 最多 100 件
        ↓
        for each salepage:
          SalepageCard (salepageCard.tsx)
            [無任何篩選]
            → 使用 ThemeCore ProductCard 渲染商品卡片
```

## 三、Shopping API 回傳的資料結構

```typescript
// Shopping API: carts/cart-extra-purchase 回傳型別
ICartExtraPurchase {
  isAppOnly: boolean;          // ← 活動是否 App 限定
  promotionId: number;
  name: string;
  typeDef: string;
  cartExtraPurchases: ICartExtraPurchaseCondtion[];
}

ICartExtraPurchaseCondtion {
  conditionId: number;
  reachPrice: number;
  reachPiece: number;
  salePageList: ICartExtraPurchaseSalePage[];
}

ICartExtraPurchaseSalePage {
  salepageId: number;
  title: string;
  picUrl: string;
  suggestPrice: number;
  price: number;
  minExtraPurchasePrice: number;
  isSoldOut: boolean;          // ← 存在，但轉換時丟失（未映射）
  soldOutActionType: string;
  order: number;
  skuList: ICartExtraPurchaseSku[];
  extraPurchaseSalepageCode: string | null;
  // ❌ 無 salePageModes（組合商品判斷依據）
  // ❌ 無 isShareToBuy
  // ❌ 無 isClosed
  // ❌ 無 sellingStartDateTime / listingStartDateTime / listingEndDateTime
}
```

## 四、篩選/檢核機制完整清單

### ✅ 有實作的篩選

| 層級 | 篩選項目 | 實作位置 | 說明 |
|------|---------|---------|------|
| 活動層 | App 限定活動 | `CartReachPriceExtraPurchaseBlock` L75-76 | Web 版排除 `isAppOnly === true` 的活動 |
| 活動層 | 活動數量上限 | `CartReachPriceExtraPurchaseBlock` L78 | `slice(0, 5)`，最多 5 檔 |
| 商品層 | 商品清單上限 | `SalepageList` L113 | `slice(0, 100)`，最多 100 件 |
| 商品層 | 整張卡片顯示 | `ReachPriceExtraPurchaseCard` L160-162 | `entityList.length === 0` 則不渲染 |

### ❌ 未實作的篩選（缺口）

| 篩選項目 | 原因 | 影響 |
|---------|------|------|
| **組合商品排除** | Shopping API 不回傳 `salePageModes`（`cartExtraPurchase.utility.ts` 預設空字串） | 組合商品可能出現在 P1 加購區 |
| **分享活動商品排除** | Shopping API 無 `isShareToBuy` 欄位 | FB 分享後購買的商品可能出現 |
| **商品頁關閉排除** | Shopping API 無 `isClosed` 欄位 | 已關閉商品頁可能出現 |
| **售完商品排除** | `isSoldOut` 在轉換時未映射到 `IPurchaseExtraSalePageEntity` | 售完商品仍顯示於卡片 |
| **販賣時間檢查** | `sellingStartDateTime` 等時間欄位皆為 `null` | 無法判斷是否在販賣期間 |

## 五、未達門檻的額外行為

當 `unReachAmount > 0`（購物車未達加購門檻）：
- 商品卡片**仍然顯示**（不隱藏）
- 點擊「我要購買」按鈕 → 彈窗：「再差 NT$XXX 才可加購」
  → 提供「前往活動」 → 導向 `/V2/PromotionEngine/Detail/{id}`
- 點擊「查看 ›」按鈕 → 導向 ExtraPurchaseList（from=cart 模式）

## 七、與 ExtraPurchaseList 頁面（位置 B）的比較

| 比較項 | P1 加購區（位置 A） | ExtraPurchaseList（位置 B） |
|--------|---------------------|---------------------------|
| **前端應用** | `CartClientApp` | `ClientApp` |
| **商品資料 API** | Shopping API → **內部呼叫 FTS (SPL)** `cart-extra-purchase` | FTS (SPL) `cart-extra-purchase` 直接呼叫 |
| **有無商品卡片** | ✅ 有（inline 商品列表） | ✅ 有（完整列表頁） |
| **組合商品排除** | ❌ 無（SPL 未排除，欄位未回傳） | ❌ 無（SPL 未排除） |
| **分享活動排除** | ❌ 無（SPL 無此欄位） | ❌ 無（SPL 無此欄位） |
| **商品關閉排除** | ❌ 無（SPL 無此欄位） | ❌ 無（SPL 無此欄位） |
| **售完商品排除** | ✅ SPL 已排除（但 isSoldOut 欄位轉換時丟失，前端無法顯示） | ✅ SPL 已排除 `IsSoldOut` |
| **篩選責任歸屬** | FTS (SPL) → Shopping API (Cart Domain) → MWeb CartClientApp | FTS (SPL) → MWeb ClientApp |
| **篩選缺口根源** | 兩條路徑**相同**：均在 FTS `BuildPromotionEntitiesAsync()` | 同左 |

## 八、重要結論

1. **P1 加購區與 ExtraPurchaseList 頁面共用同一個 FTS (SPL) 篩選層**：Shopping API (Cart Domain) 內部對每個符合活動呼叫 FTS `salepage/cart-extra-purchase`，商品篩選邏輯與 ExtraPurchaseList 完全相同（根據型別結構一致推斷，Cart Domain repo 未包含在本 repo）
2. **兩條路徑的篩選缺口根源相同**：FTS `BuildPromotionEntitiesAsync()` 只排除 `IsSoldOut`，未排除 `IsSalePageBundle`、`IsShareToBuy`、`IsClosed`
3. **MWeb 前端層的篩選差異**：
   - ExtraPurchaseList 頁：可透過 MWeb BLV2 增加額外篩選（有完整商品欄位）
   - P1 加購區：Shopping API 不回傳 `salePageModes`、`isShareToBuy`、`isClosed` 等欄位，MWeb 前端**無法**加入這些篩選
4. **`isSoldOut` 特殊狀況**：FTS 已在 SPL 層排除售完商品，但 `cartExtraPurchase.utility.ts` 未映射 `isSoldOut` 欄位，若 Shopping API 回傳了售完商品（萬一 Shopping API 未透過同一 FTS 篩選），前端卡片亦無法顯示售完狀態
