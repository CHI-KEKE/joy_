# 01 — 需求說明

> User Story [#596793](https://91appinc.visualstudio.com/MoneyLogistic/_workitems/edit/596793)  
> Sprint：MoneyLogistic \ Sprint 192　｜　負責人：Janet Chou

---

## 一句話說明

確保「滿額加價購」活動的加購商品在購買時，系統自動排除其原有的「買就送」贈品設定，防止消費者在享有加購優惠的同時又額外獲得贈品。

---

## 業務背景

### 痛點情境

1. 營運人員在 OSM → 贈品管理 → 新增贈品，將**商品 A** 設定為「買就送」活動的主商品（買 A 送 B）
2. 行銷人員同時將**商品 A** 設定為某「滿額加價購」活動的加購品
3. 消費者達成滿額門檻後，以**優惠加購價**買了商品 A
4. 系統又自動送出商品 B → **品牌同時承擔加購折扣 + 贈品成本**（雙重損失）

### 目的

當系統檢核到商品是以「加購品」身分（`OptionalTypeDef = CartReachPriceExtraPurchase`）進入購物車時，強制阻斷該商品對應的「買就送」觸發邏輯。

---

## 驗收條件（Acceptance Criteria）

### AC1：驗證「加購品」身分之贈品排除

- **假設**：商品 A 已在後台設定為某「買就送」活動的主商品
- **操作**：消費者達成滿額門檻後，從購物車加購區選購商品 A
- **預期**：結帳過程中，系統**不應**為該筆加購商品 A 分配任何贈品庫存或紀錄

### AC2：驗證「一般商品」身分之贈品正常運作

- **假設**：同一訂單中，消費者除了加購商品 A，也以**一般價格**購買了另一件商品 A
- **預期**：
  - 以「一般價格」購買的商品 A → **正常觸發**買就送、顯示並分配贈品 ✅
  - 以「加購品價格」購買的商品 A → **不觸發**買就送 ✅
  - 系統依據加入來源（加購 vs 一般）精準區分

### AC3：驗證購物車與結帳頁面 UI

- 加購區塊選購商品 A 時，**不應出現**買就送提示文字或贈品預覽圖
- 結帳試算時，贈品清單**不應包含**因加購商品 A 而產生的贈品

---

## 買就送設定路徑（後台）

```
OSM → 贈品管理 → 新增贈品
  → 設定商品贈品(買就送)
  → 贈品管理：查詢「商品贈品(買就送)」
  → 選擇買就送商品「加至商品頁」
  → 選擇要贈送的商品
```

對應 DB 資料表：`SalePageGift`（主檔）、`SalePageGiftSlave`（明細）
