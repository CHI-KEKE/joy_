# 02 識別機制

## 排除條件欄位來源

SPL `CartExtraPurchaseService` 透過 `GetSalepageListByIdsResponse` 取得商品資訊（`SalePageListingHttpClient.GetSalepageListByIdsAsync()`）。

### 欄位確認表

| 排除條件 | 欄位名稱 | 型別 | 存在於 Response？ | 備註 |
|---------|---------|------|----------------|------|
| 組合商品 | `IsSalePageBundle` | bool（計算屬性） | ✅ 存在 | `SalePageModes` 含 `'C'` 時為 true |
| 商品頁關閉 | `IsClosed` / `StatusDef` | bool / string | ✅ 存在 | StatusDef == "IsClosed" |
| 販售時間無效 | `SellingStartDateTime` / `SellingEndDateTime` | DateTime | ✅ 存在 | Story #596792 已修正 |
| 售完 | `IsSoldOut` | bool | ✅ 存在 | **已排除**（現有邏輯） |
| FB 分享後購買 | `IsShareToBuy` | - | ❌ **不存在** | 需確認替代方案 |

### IsSalePageBundle 計算邏輯（SPL 內部）
```
IsSalePageBundle = SalePageModes.Contains('C')
```
SPL response 中為已計算完成的布林值，可直接使用。

### IsClosed 判斷邏輯（SPL 內部）
`StatusDef` 欄位的可能值：
- `"Published"` → 已上架（開啟）
- `"IsClosed"` → 已關閉
- `"Draft"` → 草稿

判斷方式：`salepageInfo.IsClosed == true` 或 `salepageInfo.StatusDef == "IsClosed"`

### IsShareToBuy 問題說明
- CartAddOns（`SalepageExtraPurchaseService`）在其使用的 response model 中有此欄位
- SPL `CartExtraPurchaseService` 使用的 `GetSalepageListByIdsResponse` 中**無此欄位**
- 可能的解決方式：
  1. 在 `GetSalepageListByIdsRequest` 加入過濾參數 `ExcludeShareToBuy = true`（需 SPL 端支援）
  2. 在 `GetSalepageListByIdsResponse` 新增 `IsShareToBuy` 欄位（需 SPL 端新增）
  3. ✅ 確認業務需求後，若 CartAddOns 基準也無此排除，則暫不處理

## CartAddOns 對照（SalepageExtraPurchaseService）

CartAddOns 使用的是 WebStore DB 直接查詢，**不經過 SPL**：
```
BLV2 PurchaseExtraService → WebStore DA → SalePageEntity
```

滿額加價購使用的是 SPL API：
```
SPL CartExtraPurchaseService → GetSalepageListByIdsAsync()
```

兩者架構完全不同，排除條件需各自實作。
