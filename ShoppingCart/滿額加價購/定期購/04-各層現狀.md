## Shopping API ✅ Pass-through，無需修改

Shopping API 對 `OptionalTypeDef` 採透傳設計，不做額外處理：

- `CartService.CartsSkuAddAsync()`：直接將 `requestEntity.OptionalTypeDef` 傳給 Cart API
- `CartCreateProcessor.ExecuteAsync()`：直接將 Cart API 回傳的 `CartEntity` 設為 `context.Data`

`SalepageEntity.IsCartExtraPurchase`（Shopping BE）宣告為 `bool IsCartExtraPurchase { get; set; }`，由 Cart API 回傳值填入，Shopping 層不需自行計算。

## 現狀風險

目前 MWeb 已傳送 `OptionalTypeDef = 'CartReachPriceExtraPurchase'`，但 Cart API 的 `GetShippingTypeDef()` 尚未降級，導致：

1. `SaleProductShippingTypeDef` 仍為 `"RegularOrder"`
2. `GetShoppingCartRegularSalePageProcessor` 進入定期購分支，呼叫外部 RegularOrder API
3. **若該商品的定期購選項 API 回傳 null**，直接拋出 `CartCreateException(SalePageRegularOrderOptionNotAvailable)`，**整個購物車頁面炸掉**
4. 即使成功，前端仍顯示「定期購」標籤與定期購選項 UI

**此為 Production BUG 風險，需優先修復。**
