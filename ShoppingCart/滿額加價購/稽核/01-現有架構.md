# 01 現有稽核架構

## OrderAuditJob 流程

```
NMQ 事件
    │
    ├─ EventName == "OrderCancelled" ──► OrderCancelAuditProcessorLayers
    │
    └─ 一般下單事件
          ├─ 從 TradesOrderGroupService 取 PayUniqueKey
          ├─ 從 Redis 載入 PayProcessContextEntity
          │   （找不到 → IsForCache=false 的最小 Entity）
          └─ 進入 OrderPlaceAuditProcessorLayers
```

## 現有 OrderPlaceAuditProcessorLayers

| # | Processor | 說明 | IsSkip 條件 |
|---|---|---|---|
| 1 | `OrderInfoProcessor` | 從 DB 補訂單資料（IsForCache=false 才執行）| 永不 Skip |
| 2 | `OrderCouponDataAuditProcessor` | 折價券資料、面額、攤提稽核 | 無折價券則 Skip |
| 3 | `CouponRedeemAuditProcessor` | 優惠券核銷狀態稽核 | 無使用任何券則 Skip |
| 4 | `SalepageExtendInfoAuditProcessor` | 商品加價購稽核 | 無加購品 ExtendInfo 則 Skip |
| 5 | `SalepageBundleAuditProcessor` | 組合商品稽核 | 無組合商品則 Skip |

## Processor 共同規則

- 每個 Processor 各自 try/catch，一個掛掉不影響其他
- `IsForCache = false` 時：大部分 Processor 直接 Skip 或 Log「尚未實作」
- 稽核結果透過 `MetricsLogEntity` + `LogInformation` 輸出，不拋例外
