# 限購數量稽核 — 邊界情境與設計決策

## 邊界情境 1：不同活動包含相同加購商品

### 情境說明

活動 A（PromotionId=100）和活動 B（PromotionId=200）都設定了 SalePageId=9001 作為加購品。

### 為什麼不會混淆

每個活動對同一商品建立的是**獨立的 SpecialPrice 記錄**，各有唯一的 `SpecialPriceId`：

```
活動 A: SalePageId=9001, SkuId=X → SpecialPriceId=501
活動 B: SalePageId=9001, SkuId=X → SpecialPriceId=601
```

購物車快照中的加購品會記錄自己的 `SpecialPriceId`，  
因此用 `SpecialPriceId + SkuId` 匹配，天然區分了不同活動的加購品，不會跨活動混計。

---

## 邊界情境 2：同活動不同門檻包含相同加購商品

### 情境說明

活動 A 的 ConditionId=1（滿100元）和 ConditionId=2（滿200元）都包含 SalePageId=9001，  
通常是不同門檻定不同加購價格。

```
ConditionId=1: SalePageId=9001 → SpecialPriceId=501, Price=50元
ConditionId=2: SalePageId=9001 → SpecialPriceId=502, Price=30元（高門檻更便宜）
```

### 為什麼不會混淆

不同門檻對同一商品的 `SpecialPriceId` 不同。  
用 `(PromotionId, ConditionId)` 分組後，各門檻只會匹配到對應 `SpecialPriceId` 的加購品，  
兩者 `Qty` 獨立計算，不互相影響。

---

## 為什麼分組鍵必須是 `(PromotionId, ConditionId)` 而不是只用 `ConditionId`

多個活動各自都可能有 `ConditionId=1`、`ConditionId=2`。  
若只用 `ConditionId` 分組，不同活動的資料會被合併：

```
❌ 錯誤分組（只用 ConditionId）：
   ConditionId=1 → 活動A的加購品 + 活動B的加購品混在一起 → actualQty 錯誤

✅ 正確分組（PromotionId + ConditionId）：
   (PromotionId=100, ConditionId=1) → 只含活動A的加購品
   (PromotionId=200, ConditionId=1) → 只含活動B的加購品
```

---

## 為什麼不能對每個加購品個別比對 LimitQty

LimitQty 是「此門檻下所有加購品**總計**能買幾件」，而非「每種商品各自能買幾件」。

```
❌ 錯誤做法（逐一比對）：
   9001 Qty=2 <= LimitQty=3 → 通過
   9003 Qty=2 <= LimitQty=3 → 通過
   總計購買了 4 件，已超過限購 3 件 → 應該 Fail 卻通過

✅ 正確做法（加總比對）：
   actualQty = 2 + 2 = 4 > LimitQty=3 → Fail
```

## LimitQty = -1 的語意

從 Promotion API `GetPromotionDetail` 實測確認：`LimitQty = -1` 代表「商家未設定限購數量，無限制」。

判斷邏輯：

```csharp
if (condition.LimitQty == -1)
{
    //// 無限制，跳過此門檻的限購稽核
    continue;
}
```

## IsCyclable 對 cartTotalPrice 的依賴

Cyclable 模式下需要「購物車主商品總金額」來計算循環次數：

```
cycles = floor(cartTotalPrice / ReachPrice)
```

**cartTotalPrice 計算方式**：購物車快照中所有 `IsCartExtraPurchase = false` 的商品 `Price × Qty` 加總。

> 注意：這是簡化計算，未嚴格依照活動的 `productScopes` 篩選主商品範圍。  
> 偏差方向為「寬鬆」（金額可能略高於活動實際計算值），不會造成誤判 Fail。

## 稽核與 UpdatedDateTime 的處理

目前 `GetPromotionDetail` 回傳的 `ExtraPurchaseRuleList` **沒有 UpdatedDateTime 欄位**，  
因此限購數量稽核**無法做「活動設定在訂單後更新 → 跳過」的容錯**。

若未來需要這個機制，需要 Promotion API 端補充 `ExtraPurchaseRuleList` 的更新時間。
