# 滿件加價購 購物車商品卡文案 — 總體路徑分析

> 活動類型：`CartReachPieceExtraPurchase`  
> 關聯 User Story：599198

---

## 概覽：四個文案位置

| 文案 | 卡片位置 | 情境 | 資料來源路徑 |
|------|---------|------|------------|
| 文案一 | 加購品商品卡 | 主商品**已達**門檻，顯示加購品門檻件數 | `MatchedProcess → PromotionRecord → CartExtraPurchasePricePairs.ConditionPiece` |
| 文案二 | 主商品商品卡 | 主商品**已達**門檻，顯示活動門檻 | `MatchedProcess → PromotionRecord → CartExtraPurchaseMaxReachCondition.ReachPiece` |
| 文案三 | 主商品商品卡 | 主商品**未達**門檻，顯示再湊件數 | `MismatchedProcess → HintInfo.LackOfPiece → LackPieceCount` |
| 文案四 | 加購品商品卡 | 主商品**未達**門檻，顯示再湊提示 | `MismatchedProcess → HintInfo.LackOfPiece + PurchasedItemPiece` |

## Engine 兩條輸出路徑

Engine 依照「是否達到門檻」走不同流程，輸出至不同的容器：

```
CartReachPieceExtraPurchase
│
├─ MatchedProcess()        → PromotionRecord         → CalculateResponse.PromotionRecordList
│  （主商品件數 >= 門檻）    （型別安全，無 JSON 問題）
│
└─ MismatchedProcess()     → DisplayHintInstruction  → CalculateResponse.PromotionInstructionList
   （主商品件數 < 門檻）     （State: object，需 JSON 序列化/反序列化）
```

---

## 詳細資料流

### ① Engine

#### MatchedProcess（已達門檻）

```csharp
yield return new PromotionRecord
{
    CartExtraPurchasePricePairs = pricePairs,
    // 每筆 pricePair:
    //   ConditionPiece = reachedConditionDict[pair.ConditionId].ReachPiece  ← 文案一

    CartExtraPurchaseMaxReachCondition = new CartExtraPurchaseCondition
    {
        ReachPiece = maxReachCondition.ReachPiece   ← 文案二
    }
};
```

#### MismatchedProcess（未達門檻）

```csharp
yield return new DisplayHintInstruction
{
    State = new HintInfo
    {
        LackOfPiece        = Math.Max(condition.ReachPiece - totalPiece, 0),  ← 文案三、四
        PurchasedItemPiece = totalPiece,                                       ← 文案四
    }
};
```

---

### ② PromotionFrontendAPI → HTTP Response JSON

- `PromotionRecordList`：型別明確，直接序列化，欄位名稱完整保留
- `PromotionInstructionList`：`State` 宣告為 `object`，以 runtime type 序列化

```json
// PromotionInstructionList 中的 State 節點
{
  "LackOfPiece": 3,           ← Engine 私有命名（非 LackPieceCount）
  "PurchasedItemPiece": 2,
  "LackOfPrice": 0,
  "PurchasedItemPrice": 0
}
```

#### CalculatePromotionProcessor 映射

**已達門檻分支**（line 753–768）：

```csharp

// 文案二來源

    = promotionRecords.First().CartExtraPurchaseMaxReachCondition;
```

**未達門檻分支**（line 669–687）：

```csharp
// 文案三、四來源（本次修復）
promotionInfo.LackPieceCount =
    promotionType == CartReachPieceExtraPurchase
        ? hintState.LackOfPiece          // ← 修復點：原本用 hintState.LackPieceCount 永遠為 0
        : promotionType != DiscountReachGroupsPiece
            ? hintState.LackPieceCount
            : hintState.Groups.Values
                .Where(x => x.GroupsLimitCycleCount == 0)
                .Sum(x => x.NextCycleLakeItemsCount);

// 文案四來源
promotionInfo.PurchasedItemPiece = hintState.PurchasedItemPiece;
```

## Bug 修復記錄

### Bug 1：文案三「再湊 0 件」

| 項目 | 內容 |
|------|------|
| 現象 | 主商品卡顯示「再湊 0 件」，件數始終為 0 |
| 根因 | Engine 回傳 `LackOfPiece`，Cart `HintInfo` 原本只有 `LackPieceCount`，JSON 反序列化靜默丟棄 |
| 修復 | `HintInfo` 新增 `LackOfPiece` 欄位；`CalculatePromotionProcessor` 針對 `CartReachPieceExtraPurchase` 改用 `hintState.LackOfPiece` |

### Bug 2：文案四無文案

| 項目 | 內容 |
|------|------|
| 現象 | 加購品卡在未達門檻時完全無提示文案 |
| 根因 | `UnreachCartExtraPurchaseDisplayTextProcessor` filter 條件 `PurchasedItemPrice > 0 && LackOfPrice > 0` 對滿件永遠為 false（滿件金額欄位都是 0） |
| 修復 | 新增 else 分支處理滿件邏輯，filter 改為 `PurchasedItemPiece > 0 \|\| LackPieceCount > 0` |
